## 一、产品定位与核心价值

### 1.1 产品概述

**核心公式**：**排级拓扑社交 + 模糊位置隐私 + 半自动数据获取**

| 决策项 | 方案 | 理由 |
|--------|------|------|
| 座位精度 | 排级（区域+排号范围） | 满足"前后排找搭子"需求，保护隐私 |
| 数据获取 | 半自动化爬取票牛数据 | 人工触发，单次提取，合规可控 |
| 社交模式 | 化名系统 + 拓扑匹配 | 降低社交压力，提升安全感 |

### 1.2 核心价值主张

```
对于：独自或与朋友一起看演唱会的年轻用户
他们想要：在演唱会现场找到附近的同好，互相照应、拍照互助
座邻是：基于座位位置的演唱会社交匹配平台
它提供：排级精度的邻居发现、安全的化名社交、后演唱会活动组织
不同于：传统社交软件（无地理位置关联）、粉丝群（无座位关联）
```

### 1.3 目标用户画像

| 用户类型 | 特征 | 核心需求 |
|----------|------|----------|
| **独行观众** | 18-30岁，独自购票 | 找搭子互相拍照、分享体验 |
| **小团体观众** | 2-3人，座位分散 | 与附近的人交换座位、扩展社交 |
| **后援会成员** | 有组织应援需求 | 快速集结、统一行动 |
| **社交探索者** | 喜欢认识新朋友 | 基于共同兴趣的轻社交 |

## 二、市场分析与竞品研究

### 2.1 市场背景

- **演唱会市场规模**：2024年中国演出市场规模超500亿元，演唱会占比超60%
- **用户痛点**：独自看演唱会占比约35%，存在强烈的社交需求
- **现有解决方案缺陷**：
  - 微博/小红书：信息分散，无法精准匹配座位附近的人
  - 粉丝群：门槛高，非粉丝难以融入
  - 现场搭讪：社交压力大，成功率低

### 2.2 竞品分析

| 竞品 | 类型 | 优势 | 劣势 | 我们的差异化 |
|------|------|------|------|--------------|
| **微博超话** | 社区 | 用户基数大 | 信息杂乱，无位置关联 | 座位级精准匹配 |
| **小红书** | 内容平台 | 内容丰富 | 被动发现，无实时性 | 实时邻居发现 |
| **大麦/猫眼** | 票务平台 | 购票入口 | 无社交功能 | 社交+票务数据打通 |
| **即刻** | 兴趣社区 | 圈子文化 | 无线下场景 | 演唱会场景垂直深耕 |

### 2.3 SWOT分析

| 优势(Strengths) | 劣势(Weaknesses) |
|-----------------|------------------|
| 排级精度平衡隐私与社交 | 冷启动需要场馆数据积累 |
| 化名系统降低社交门槛 | 依赖演唱会档期，非高频需求 |
| 垂直场景，用户意图明确 | 需要与票务平台合作获取数据 |

| 机会(Opportunities) | 威胁(Threats) |
|---------------------|---------------|
| 演唱会市场持续增长 | 票务平台可能自建社交功能 |
| Z世代社交需求旺盛 | 数据合规风险 |
| 后演唱会经济蓝海 | 用户留存率低（场景性产品） |

## 四、数据模型设计

### 4.1 核心原则

放弃PostGIS，使用**拓扑关系表**存储排级邻近关系，降低技术复杂度。

### 4.2 数据库表结构

```sql
-- ============================================
-- 场馆基础表
-- ============================================
CREATE TABLE venues (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name VARCHAR(100) NOT NULL,           -- 梅赛德斯奔驰文化中心
    city VARCHAR(20) NOT NULL,
    address TEXT,
    capacity INT,                          -- 场馆容量
    source VARCHAR(50) DEFAULT 'manual',  -- 'piaoniu_crawled' | 'manual'
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

-- ============================================
-- 区域定义表（爬取或手动录入）
-- ============================================
CREATE TABLE venue_areas (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    venue_id UUID NOT NULL REFERENCES venues(id) ON DELETE CASCADE,
    area_code VARCHAR(20) NOT NULL,       -- "A区"
    area_name VARCHAR(50) NOT NULL,       -- "看台A区"
    area_type VARCHAR(20) CHECK (area_type IN ('floor', 'stand', 'vip')),
    svg_path TEXT,                        -- SVG轮廓路径（票牛直接复用）
    total_rows INT,                       -- 总排数
    center_x FLOAT,                       -- 热力图坐标（相对0-100）
    center_y FLOAT,
    created_at TIMESTAMP DEFAULT NOW()
);

-- ============================================
-- 排号定义表（核心）
-- ============================================
CREATE TABLE rows (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    area_id UUID NOT NULL REFERENCES venue_areas(id) ON DELETE CASCADE,
    row_number INT NOT NULL,              -- 数字排号（1,2,3...）
    row_label VARCHAR(10),                -- 显示名（"1排"、"AA排"）
    tier INT DEFAULT 1,                   -- 楼层
    avg_seats INT,                        -- 平均每排座位数（估算）
    UNIQUE(area_id, row_number)
);

-- ============================================
-- 演唱会表
-- ============================================
CREATE TABLE concerts (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    title VARCHAR(200) NOT NULL,          -- 演唱会名称
    artist VARCHAR(100),                  -- 艺人
    venue_id UUID REFERENCES venues(id),
    start_time TIMESTAMP,                 -- 开场时间
    status VARCHAR(20) DEFAULT 'upcoming', -- upcoming/ongoing/completed
    poster_url TEXT,                      -- 海报URL
    created_at TIMESTAMP DEFAULT NOW()
);

-- ============================================
-- 用户座位绑定表（业务核心表）
-- ============================================
CREATE TABLE user_concert_seats (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL,
    concert_id UUID NOT NULL REFERENCES concerts(id) ON DELETE CASCADE,
    area_id UUID NOT NULL REFERENCES venue_areas(id),
    row_id UUID NOT NULL REFERENCES rows(id),
    row_range_start INT,                  -- 排范围开始（如5）
    row_range_end INT,                    -- 排范围结束（如7）
    seat_number VARCHAR(10),              -- 选填：座位号
    visibility VARCHAR(20) DEFAULT 'row_only' CHECK (visibility IN ('public', 'row_only', 'area_only', 'private')),
    pseudo_name VARCHAR(50),              -- 系统生成化名（如"追光者A3"）
    social_tags TEXT[],                   -- ["带相机", "可拼车"]
    virtual_x FLOAT,                      -- 区域内随机坐标（0-100）
    virtual_y FLOAT,                      -- 用于散点图展示
    identity_mode VARCHAR(20) DEFAULT 'pseudo' CHECK (identity_mode IN ('transparent', 'pseudo', 'ghost', 'anonymous')),
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW(),
    UNIQUE(user_id, concert_id)
);

-- ============================================
-- 拓扑邻近关系表（预计算，解决性能）
-- ============================================
CREATE TABLE seat_proximity (
    user_a_id UUID NOT NULL,
    user_b_id UUID NOT NULL,
    concert_id UUID NOT NULL REFERENCES concerts(id) ON DELETE CASCADE,
    distance_text VARCHAR(50),            -- "同排"、"前2排"、"后3排"
    distance_score INT,                   -- 0=同排，1=相邻，2=隔1排...
    can_interact BOOLEAN DEFAULT true,    -- 是否允许互动（如3排内）
    created_at TIMESTAMP DEFAULT NOW(),
    PRIMARY KEY (user_a_id, user_b_id, concert_id)
);

-- ============================================
-- 视角照片表
-- ============================================
CREATE TABLE view_photos (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL,
    concert_id UUID NOT NULL REFERENCES concerts(id) ON DELETE CASCADE,
    area_id UUID NOT NULL REFERENCES venue_areas(id),
    row_id UUID REFERENCES rows(id),
    photo_url TEXT NOT NULL,
    rating VARCHAR(20) CHECK (rating IN ('excellent', 'good', 'partial', 'blocked')),
    description TEXT,
    is_delayed BOOLEAN DEFAULT false,     -- 是否延时发布
    publish_time TIMESTAMP,               -- 计划发布时间
    created_at TIMESTAMP DEFAULT NOW()
);

-- ============================================
-- 后演唱会活动表
-- ============================================
CREATE TABLE post_concert_activities (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    concert_id UUID NOT NULL REFERENCES concerts(id) ON DELETE CASCADE,
    creator_id UUID NOT NULL,
    activity_type VARCHAR(20) CHECK (activity_type IN ('dinner', 'ktv', 'carpool', 'other')),
    title VARCHAR(100) NOT NULL,
    description TEXT,
    location TEXT,
    start_time TIMESTAMP,
    max_participants INT DEFAULT 10,
    status VARCHAR(20) DEFAULT 'open',
    secret_code VARCHAR(50),              -- 接头暗号
    created_at TIMESTAMP DEFAULT NOW()
);

-- ============================================
-- 活动参与者表
-- ============================================
CREATE TABLE activity_participants (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    activity_id UUID NOT NULL REFERENCES post_concert_activities(id) ON DELETE CASCADE,
    user_id UUID NOT NULL,
    status VARCHAR(20) DEFAULT 'joined',  -- joined/confirmed/cancelled
    joined_at TIMESTAMP DEFAULT NOW(),
    UNIQUE(activity_id, user_id)
);
```

### 4.3 邻近度计算逻辑

```python
# 伪代码：用户录入座位时异步预计算
def calculate_proximity(user_id: UUID, concert_id: UUID, area_id: UUID, row_num: int):
    """
    计算用户与同一演唱会同区域其他用户的邻近关系
    """
    # 查找同场同区域的其他用户
    neighbors = query("""
        SELECT user_id, row_number FROM user_concert_seats 
        WHERE concert_id = %s AND area_id = %s AND user_id != %s
    """, [concert_id, area_id, user_id])
    
    proximity_records = []
    
    for neighbor in neighbors:
        row_diff = abs(neighbor.row_number - row_num)
        
        # 排级距离分级
        if row_diff == 0:
            dist_text, score = "同排", 0
            can_interact = True
        elif row_diff <= 2:
            dist_text = f"前{row_diff}排" if neighbor.row_number < row_num else f"后{row_diff}排"
            score = row_diff
            can_interact = True
        elif row_diff <= 5:
            dist_text, score = "附近", 5
            can_interact = True
        else:
            continue  # 超过5排不建立关系
        
        # 双向写入
        proximity_records.append({
            'user_a_id': user_id,
            'user_b_id': neighbor.user_id,
            'dist_text': dist_text,
            'score': score,
            'can_interact': can_interact
        })
        
        proximity_records.append({
            'user_a_id': neighbor.user_id,
            'user_b_id': user_id,
            'dist_text': f"后{row_diff}排" if neighbor.row_number < row_num else f"前{row_diff}排",
            'score': score,
            'can_interact': can_interact
        })
    
    # 批量插入
    batch_insert_seat_proximity(proximity_records)
```

## 五、隐私与安全设计

### 5.1 四级身份体系

| 模式 | 身份显示 | 座位精度 | 可交互性 | 适用人群 |
|------|----------|----------|----------|----------|
| **透明模式** | 真实昵称+头像 | 精确到排座号 | 完全开放 | 社交达人、后援会 |
| **化名模式**（推荐） | 系统生成化名+虚拟头像 | 散点相对位置 | 可打招呼，需验证后解锁 | 大多数用户 |
| **幽灵模式** | 仅显示"附近有人" | 仅显示"同区域" | 仅接收邀请，不可主动发起 | 社恐、测试用户 |
| **完全匿名** | 对他人不可见 | 仅系统后台记录 | 无法被匹配，仅浏览他人 | 纯工具使用者 |

### 5.2 化名生成逻辑

```python
def generate_pseudo_name(area_code: str, user_id: UUID) -> str:
    """
    生成用户化名
    - 形容词 + 名词 + 座位区号哈希
    - 同一演唱会内保持一致
    - 不可逆加密存储
    """
    adjectives = ["炽热", "追光", "星辰", "梦想", "闪耀", "璀璨", "流光"]
    nouns = ["星辰", "旅人", "追梦者", "光年", "音符", "旋律", "节拍"]
    
    # 基于area_code和user_id生成确定性哈希
    hash_input = f"{area_code}:{user_id}"
    hash_val = hashlib.md5(hash_input.encode()).hexdigest()
    
    adj_idx = int(hash_val[0:2], 16) % len(adjectives)
    noun_idx = int(hash_val[2:4], 16) % len(nouns)
    suffix = hash_val[4:6].upper()
    
    return f"{adjectives[adj_idx]}{nouns[noun_idx]}_{suffix}"
```

### 5.3 权限矩阵

| 操作 | 透明模式 | 化名模式 | 幽灵模式 | 完全匿名 |
|------|----------|----------|----------|----------|
| 被搜索发现 | ✓ 精确 | ✓ 模糊区域 | ✓ 仅计数 | ✗ 不可见 |
| 接收打招呼 | ✓ | ✓ | ✓（仅接收） | ✗ |
| 查看他人精确座位 | 双向可见 | 需对方同意 | 不可见 | 不可见 |
| 发起活动邀请 | ✓ | ✓（显示化名） | ✗ | ✗ |
| 上传视角照片 | 实名水印 | 化名水印 | 无水印 | 无水印 |

### 5.4 数据安全措施

| 数据类型 | 处理方式 |
|----------|----------|
| 照片EXIF | 上传即剥离GPS信息，前端添加化名水印 |
| 消息存储 | Redis存储，TTL 7天自动物理删除 |
| 日志脱敏 | user_id使用HMAC-SHA256哈希处理 |
| 座位映射 | 化名与真实用户ID映射表加密存储 |

### 5.5 内容审核方案

| 项目 | 方案 |
|------|------|
| 技术栈 | 腾讯云 TMS / 百度 AI 审核 |
| 策略 | 先发后审（延迟<500ms），违规立即撤回 |
| 关键词库 | 黄暴词 + "出票"/"转票"/"加钱"等票务交易词 |
| 人工审核 | 仅针对被多次举报的群聊，24小时记录 |

| 合规项 | 实现方案 |

### 6.4 核心算法

#### 6.4.1 虚拟坐标生成算法

```python
def generate_virtual_coordinates(area_id: UUID, user_id: UUID) -> Tuple[float, float]:
    """
    生成区域内虚拟坐标（用于散点图展示）
    - 基于area_id和user_id生成确定性随机坐标
    - 确保同一用户在同一区域内坐标固定
    - 坐标范围 0-100
    """
    # 使用哈希生成确定性随机数
    hash_input = f"{area_id}:{user_id}"
    hash_val = hashlib.sha256(hash_input.encode()).hexdigest()
    
    x = (int(hash_val[0:8], 16) % 10000) / 100
    y = (int(hash_val[8:16], 16) % 10000) / 100
    
    return (x, y)
```

#### 6.4.2 邻居匹配算法

```python
def find_neighbors(user_id: UUID, concert_id: UUID, max_distance: int = 3) -> List[Neighbor]:
    """
    查找附近邻居（基于预计算的拓扑关系表）
    """
    neighbors = query("""
        SELECT 
            sp.user_b_id,
            sp.distance_text,
            sp.distance_score,
            ucs.pseudo_name,
            ucs.social_tags,
            ucs.virtual_x,
            ucs.virtual_y
        FROM seat_proximity sp
        JOIN user_concert_seats ucs ON sp.user_b_id = ucs.user_id
        WHERE sp.user_a_id = %s 
          AND sp.concert_id = %s
          AND sp.distance_score <= %s
          AND sp.can_interact = true
        ORDER BY sp.distance_score ASC
    """, [user_id, concert_id, max_distance])
    
    return neighbors
```

### 12.3 空状态（Empty State）规范

#### 12.3.1 组件结构

1. 图形区：票根撕裂/聚光灯熄灭风格图标，建议 `96rpx~120rpx`。
2. 标题：`H2`（16px / 600），一句话说明当前状态。
3. 描述：`Body`（14px / 400），给下一步引导。
4. 行动按钮：主按钮优先（如“去其他城市看看”）；次操作可文字链接（如“刷新”）。

#### 12.3.2 场景文案（统一中文）

| 场景 | 标题 | 描述 | 主按钮 |
|------|------|------|--------|
| 首页无演出 | 当前城市还没有演出 | 切换城市，看看别处正在发生的现场 | 去热门城市 |
| 订阅为空 | 你还没有订阅提醒 | 订阅开售后，我们会第一时间通知你 | 去订阅演出 |
| 明日抢票为空 | 明日暂无开售场次 | 可以先订阅心愿清单，开售前自动提醒 | 去订阅页 |
| 搭子模块为空 | 附近还没人发起搭子 | 你可以成为第一个发起人，先喊一嗓子 | 发布寻搭子 |
| 网络异常 | 网络开小差了 | 检查网络后重试，我们会保留你的浏览位置 | 重新加载 |

#### 12.3.3 布局与间距

- 容器：垂直居中，最小高度建议 `360rpx`。
- 图标与标题间距：`24rpx`。
- 标题与描述间距：`12rpx`。
- 描述与按钮间距：`24rpx`。
- 整体左右安全边距：`32rpx`（16px）。

### 12.4 骨架屏（Skeleton）规范

#### 12.4.1 骨架形态模板

| 页面 | 骨架模块 | 建议高度 |
|------|----------|----------|
| 首页列表 | 海报块 + 两行文本 + 底部标签条 | 海报 `420rpx`，文本行 `24rpx/20rpx` |
| 详情页 | Hero 海报 + 票根信息卡 + 平台按钮条 + 座位图区块 | Hero `520rpx`，座位图 `500rpx` |
| 明日抢票/订阅 | 时间轴节点圆点 + 右侧卡片块 | 卡片 `220rpx` |
| 管理页 | 表单输入条 + 上传占位框 + 按钮占位 | 输入条 `88rpx`，上传框 `240rpx` |

#### 12.4.2 动效规则

- 流光方向：从左到右。
- 单周期：`1200ms`，`linear` 无限循环。
- 入场动画：从 `translateY(20px)` 到 `0`，透明度 `0 -> 1`，时长 `300ms`，曲线 `cubic-bezier(0.2, 0.8, 0.2, 1)`。
- 分层出现：大模块延迟 `60ms` 级联，避免“同时闪现”。

#### 12.4.3 视觉限制

- 骨架圆角必须匹配真实组件（卡片 `16px`，按钮 `12px`）。
- 骨架色差控制在低对比区间，避免比真实内容更抢眼。
- 单屏骨架数量不超过真实内容上限，防止误导用户预期。

### 12.5 加载态（Loading State）规范

#### 12.5.1 三层加载体系

| 层级 | 触发场景 | 形态 |
|------|----------|------|
| 轻量加载 | 列表分页、筛选切换 | 列表底部文案 + 三点跳动 |
| 区块加载 | 座位图重算、卡片局部刷新 | 模块内蒙层 + 迷你骨架 |
| 页面加载 | 首次进入详情页 | 全页骨架 + 轻蒙层，禁止按钮重复触发 |

#### 12.5.2 反馈节奏

- `0~300ms`：不展示加载器，避免闪烁。
- `300~1500ms`：展示轻量加载（点状或细线进度）。
- `>1500ms`：升级为“文案提示 + 可取消/重试”。
- `>8000ms`：进入弱失败提示（非阻断），给“重试”按钮。

#### 12.5.3 文案规范（统一中文）

- 加载中：`正在为你同步最新演出信息...`
- 提交中：`正在提交，马上就好...`
- 座位图计算中：`正在定位区域视角，请稍候...`
- 超时提示：`加载有点慢，点这里重试`

### 12.6 交互与可访问性

- 所有状态模块支持 `:active` 压感反馈：`transform: scale(0.98)`。
- 座位图分区点击后触发一次透明度闪烁 + `wx.vibrateShort`。
- 文字对比度：正文至少满足 WCAG AA（建议对比度 >= 4.5:1）。
- 提供低动效开关：当用户开启“减少动态效果”时，关闭流光动画，仅保留静态骨架。
